Chapter 5: Implementation

The design in Chapter 4 became a working prototype through plain HTML, JavaScript, and a Vite dev server. DOM event listeners capture user behaviour on a simulated phishing page; a static dashboard displays risk scores and event timelines. The methodology (Chapter 3) called for iterative phases, and that is how development proceeded.

5.1 Implementation Overview and Methodology Mapping

Each phase was finished and tested before moving on. The order was: phishing simulation and event capture first, then the dashboard for risk display and the event timeline, then wiring everything up to the API. No formal sprints were used. Early validation and small tweaks between phases turned out to be enough.

Runtime monitoring (3.4.1) is DOM event listeners on a simulated phishing page with a browser like interface. Events are sent to the backend via RESTful POST requests at /backend/api/behavior-events (3.4.2). Risk reporting (3.4.4) is a static dashboard that loads events and analysis and shows risk scores, classifications, and factors. The backend handles storage and behaviour analysis; the client simply calls the endpoints from Chapter 4.

5.2 Project Structure and Technology Stack

Table 5.1 summarises the technologies used in the implementation and the justification for each choice.

Table 5.1 Technologies used in implementation and their justification

| Technology | Purpose | Justification |
|------------|---------|---------------|
| HTML, CSS, JavaScript | Phishing sim, dashboard, test page | No build step; static files load directly; matches design for lightweight client (NFR1) |
| Vite | Dev server and proxy | Serves static assets; proxies /backend to Spring Boot on port 8080; simple setup |
| Spring Boot | Backend API | RESTful endpoints for event storage and analysis; industry standard for Java web services |
| H2 | Database | Embedded, in-memory or file-based; no separate DB server; sufficient for prototype (FR3) |
| Chrome | Testing environment | Widely used; good support for DOM APIs and DevTools for debugging |
| sendBeacon / fetch | Event transmission | Reliable delivery on tab close (sendBeacon); fetch with keepalive as fallback |

All client side assets are static and served by Vite. HTML, CSS, and JavaScript sit in the public directory. The phishing sim, dashboard, and test events page are standalone files with scripts inline. No build step. They load as written.

vite.config.ts sets up a dev proxy: paths starting with /backend are forwarded to http://localhost:8080 with the prefix stripped. A call to /backend/api/behavior-events becomes http://localhost:8080/api/behavior-events. This lets the frontend talk to a separate backend. The pages use plain JavaScript, no TypeScript, and no ML or analytics libraries.

5.3 Phishing Simulation Implementation

5.3.1 Interface and Layout

One HTML file holds the phishing sim, with inline CSS and JavaScript. It mimics a Google new tab in dark theme (background 202124, accents 8ab4f8, 9aa0a6). Layout: tab bar ("New Tab", "+"), nav bar (back, forward, refresh), address bar with "Search Google or type a URL", and a "Session dashboard" button. showPage(id) toggles the .active class to switch views (display: block for active, display: none for others).

The home page (id page home) has a large "Google" heading, a search box, and six shortcut tiles: Gmail, Outlook, Drive, Maps, YouTube, Docs. Each has a data page attribute (gmail, outlook, etc.) for navigation. Clicking a shortcut shows the corresponding page (e.g. page gmail) with a sign in style prompt. Typing a query and pressing Enter in the search or address bar brings up search results. Seven variants exist (search, search outlook, search gmail, and so on), each with a search input and fake results; six destination pages (gmail, outlook, drive, maps, youtube, docs) show sign in prompts. The footer link "View session dashboard" and nav bar "Session dashboard" both route to the dashboard. Page title stays "New Tab" for the browser like illusion.

5.3.2 Session Identification and Event Payload Structure

On load, the session id is generated with Math.random().toString(36).slice(2). Every event in that session includes it. performance.now() tracks time since navigation for timing. Each API payload is { sessionId, timestamp, event }, with event a type specific object (e.g. { type: "click", element: { tag, id } }).

5.3.3 Event Capture Logic

Native DOM listeners with capture phase (true) run first. Captured events:

First interaction: The first click or keydown emits first_interaction. A closure variable marks the first occurrence; later ones are ignored. Payload: interactionType (click or keydown) and timeSincePageLoadMs.

Click: Every click emits a click event. A helper walks up to the nearest a, button, input, or textarea. When found, the payload gets element with tag and id (id can be null). If the target is not inside something interactive, element is null.

Keydown: Every keypress emits keydown. Key values are not stored to avoid logging sensitive input.

Scroll depth: A window scroll listener tracks the max depth. Content uses overflow y auto. Depth = (scrollY + innerHeight) / scrollHeight, capped at 1. When the stored max increases, a scroll_depth event goes out with maxDepthPercent (0–100). The listener uses { passive: true } to avoid jank.

Session end: beforeunload sends session_end with maxScrollDepthPercent and timeOnPageMs (performance.now() minus navStart, rounded). That way a final event is sent even when the user closes the tab. sendBeacon is used when available since it survives unload.

5.3.3.1 Event Payload Reference

Full API payload: { sessionId, timestamp, event }. Inner event shapes:

first_interaction: { type: "first_interaction", interactionType: "click" or "keydown", timeSincePageLoadMs: number }

click: { type: "click", element: { tag: string, id: string or null } or null }

keydown: { type: "keydown", element: { tag: string, id: string or null } or null }

scroll_depth: { type: "scroll_depth", maxDepthPercent: number }

session_end: { type: "session_end", maxScrollDepthPercent: number, timeOnPageMs: number }

5.3.4 Event Transmission

POST to /backend/api/behavior-events. The send function builds { sessionId, timestamp, event }, stringifies it, and chooses: sendBeacon (if available) with a JSON Blob and type "application/json", or fetch with POST, Content-Type application/json, keepalive: true. keepalive helps the request complete on unload. No retry; failures are swallowed via an empty .catch so the client stays light.

5.3.5 Routing and Simulated Navigation

Search and address bar go through routeSearch() and showPage(). routeSearch maps queries: outlook, microsoft, office, live.com, hotmail → search outlook; gmail, google mail → search gmail; drive → search drive; maps → search maps; youtube, video → search youtube; docs → search docs; else generic search. The home search box and each search results page call showPage(routeSearch(query)) on Enter.

The address bar (id address) on Enter: "dashboard", "session", or "session data" calls goToDashboard(). Else: outlook, microsoft, office, live, hotmail → page outlook; mail or gmail → page gmail; drive, maps, youtube, video, docs → their pages. Anything else falls back to routeSearch. goToDashboard stores sessionId in sessionStorage under lastSessionId and redirects to the dashboard with the session id as a query param for preselection. Scripts run in an IIFE to avoid globals.

5.4 API Integration Points

The client hits the API from Chapter 4. The phishing sim POSTs events to /backend/api/behavior-events with Content-Type application/json. The dashboard GETs events from the same path (array) and analysis from /backend/api/behavior-events/sessions/{sessionId}/analysis. Analysis: riskScore (number), classification (e.g. "Normal", "Moderate risk", "Suspicious"), factors (array of strings). Events: sessionId, eventType (first_interaction, click, keydown, scroll_depth, session_end), receivedAt, payloadJson.

5.5 Dashboard Implementation

5.5.1 Layout and Session Selection

The dashboard is a static HTML file with embedded CSS and JavaScript. Light theme (background f8f9fa) contrasts with the sim's dark theme. Page contents: title "Session Dashboard", subtitle "Behaviour data captured from the phishing sim", session selector dropdown (hidden when there is only one or zero sessions), risk section, stats row, event timeline. "Back to phishing sim (continue browsing)" links back.

getSession() picks the session: URL query param session first, then sessionStorage lastSessionId. getSessionsFromEvents() groups by sessionId, counts events, records last receivedAt, sorts by last activity. The dropdown only fills when there are two or more sessions; options show the first 8 chars of the id, event count, last activity. If the URL points to a session not yet in the list, it is prepended. With no session specified but events present, the most recent wins.

5.5.2 Data Loading and Risk Display

On load the dashboard fetches GET /backend/api/behavior-events. Success → render(events). Failure → stats area shows "Failed to load events. Is the backend running?" and a console warning.

For the selected session it fetches GET /backend/api/behavior-events/sessions/{sessionId}/analysis. The result is rendered as a risk card: "Post session risk analysis", riskScore with "% risk", classification, bullet list of factors. Classification is lowercased and spaces replaced for a valid CSS class: "suspicious" → suspicious (red dc3545), "moderate" → moderate (amber ffc107), else normal (green 28a745). Analysis fail → risk section cleared, console warning.

Events are filtered by sessionId. If the session has none, the last 50 from the full list (events.slice(-50)) are shown instead.

5.5.3 Event Timeline and Statistics

Stats from session events: total count, clicks (eventType === "click"), keypresses (eventType === "keydown"), scroll_depth count. Time to first interaction from the first first_interaction: parse payloadJson and read timeSincePageLoadMs, shown in its own stat.

Timeline: events newest first. Each row: eventType, receivedAt (toLocaleTimeString()), detail from getEventDetail(). scroll_depth → "Scroll: X%"; click → "Element: TAG" from event.element.tag or empty; keydown → "Key press"; rest blank.

5.5.4 Error Handling and Edge Cases

Events fetch fail → catch updates stats with an error. Analysis fetch fail → risk section emptied; timeline and stats still show if events loaded. Changing session in the dropdown runs navigateToSession(), which updates the URL with history.replaceState and rerenders. Session selector hidden when sessions.length <= 1.

5.6 Testing During Implementation

Testing happened as development went along, not only at the end. After each phase (phishing sim, event transmission, dashboard, API integration) manual black box tests were run.

5.6.1 Test Scenarios and Validation

Manual tests followed the methodology. Clicks, typing, and scrolling on the phishing sim, then verification:

Event capture: Clicks, keypresses, and scrolls showed up in the dashboard timeline. First interaction timing was verified by acting fast after load versus waiting.

Event transmission: The sim sends events via sendBeacon or fetch. The test events page was used to validate transmission in isolation.

Dashboard: Risk card, factors, and timeline render when data is present. The session selector appears with multiple sessions; switching updates the view.

5.6.2 Test Support Artefacts

A separate test events page exercises capture and transmission. It uses fetch instead of sendBeacon so the response can be inspected: r.ok → green "Event sent" plus event type; failure → red status. It has a "Click me" button (click with element.tag "BUTTON"), a text input (keydown with element.tag "INPUT"), and a scrollable div (id scroll area) with 20 lines. Scroll depth comes from the div's scrollTop, clientHeight, scrollHeight, unlike the sim which uses window metrics. The test page sends first_interaction, click, keydown, scroll_depth but not session_end (no beforeunload). This setup makes it easy to confirm each event type reaches the API without running the full sim. A link points to /api/behavior-events for manual inspection.

5.7 Chapter Summary

PhishShield AI was built as a phishing sim with DOM event capture and a static dashboard. The client sends events to and fetches analysis from the Chapter 4 API. Phases: phishing sim and event capture, dashboard, API integration, with manual black box testing after each. The result matches the design and provides the client side prototype evaluated in the next chapter.
